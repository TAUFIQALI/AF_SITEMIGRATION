Section 01

WTF is WordPress (for Node/React devs)

WordPress is not a framework you install with npm. Think of it as a monolithic PHP CMS that has its own runtime, database layer, template engine, and plugin system — all baked into one process.

WordPress Architecture → Your Mental Model

Browser
User visits example.com → Apache/Nginx routes to index.php
WordPress Core
Boots up, reads wp-config.php, connects MySQL, loads active theme + plugins. Think of it as Express middleware that auto-runs.
Theme
Templates like front-page.php = your route handlers + JSX combined. The filename IS the route. No router config needed.
Plugins
PHP files in wp-content/plugins/ = npm packages that hook into WP lifecycle. Your plugin lives here.
💡

The Single Most Important WordPress Concept

WordPress uses a Template Hierarchy — it picks which PHP file to render based on the URL. For the homepage, it looks for front-page.php first, then home.php, then index.php. Your plugin's job is to find which file renders the homepage, parse it, and rewrite it.

🗂️

There's No Frontend/Backend Split

WordPress is server-rendered PHP. front-page.php is both your "component" AND your "API route." It queries data, loops through it, and outputs HTML — all in the same file. No fetch(), no API, no JSON responses.

🗄️

The Database is Abstracted Away

You never write SQL. WordPress gives you functions: get_posts(), get_post_meta(), update_post_meta(). These read/write to wp_posts and wp_postmeta tables. Think of it as a built-in ORM with a terrible API.

🔌

WP-CLI = Your Node CLI

wp is a command-line tool to control WordPress. You register custom CLI commands with PHP classes. Your plugin exposes commands like wp acf-wiring scan. Think of it as Commander.js / yargs but in PHP.

📦

ACF = Headless CMS Fields

Advanced Custom Fields (ACF) is a plugin that adds custom data fields to WordPress posts. Think of it as a form builder for the database — you define fields (text, image, repeater, etc.), and editors fill them in the admin UI. Your plugin creates these field definitions and populates them with extracted data.

The Golden Rule: Never Break a Live Site

The rewritten front-page.php has an if/else fallback. If ACF data exists → render new way. If not → include the backup .bak file (the original). This means the site literally cannot break. Even if your plugin seeds garbage data, clearing the data restores the original site instantly.

Phase-by-Phase Breakdown

Phase CLI Command What It Does (in Node Terms) Input → Output
P1 Detect wp acf-wiring homepage Finds front-page.php in the theme like a file resolver. Checks WP options DB for static front page setting. Returns which PHP file renders the homepage. Theme path → homepage-assessment.json
P2 Parse wp acf-wiring homepage (same cmd) Reads the PHP file as a string. Uses regex + HTML parsing to split it into sections (hero, services, etc). Extracts every piece of content: headings, images, URLs, phones, addresses. PHP file → homepage-scan.json with section map
P3 Schema wp acf-wiring schema deploy Registers ACF field groups and CPTs by calling WordPress APIs. Like running Prisma migrate — it creates the "tables" (field definitions) that data will live in. Schema definitions → Database (ACF field groups registered)
P4 Rewrite wp acf-wiring apply-homepage Backs up the original PHP file. Generates a new front-page.php with a loop + partials. Like converting a hardcoded React component into a data-driven one with .map(). Original PHP → .bak + new templates + partials
P5 Seed wp acf-wiring seed-homepage Takes the extracted content from P2 and writes it into ACF fields via update_post_meta(). Like running a database seed script. Also downloads images to WP media library. homepage-scan.json → Database (ACF field values populated)
P6 Verify wp acf-wiring verify-homepage Reads the seeded data back, compares against the original scan. Checks: section count, content match, image existence, link validity. Like a test suite that asserts content parity. DB data + scan JSON → verification-report.json (PASS/FAIL)
P7 QA — Human opens staging URL + live URL side by side. Checks visual match, clicks links, tests admin editor. Not your problem — you just need to give them a PASS from P6. Staging URL → APPROVED / NEEDS FIX
P8 Deploy Git merge + deploy Merge branch to main. Kinsta auto-deploys. Flush caches. Smoke test. .bak stays for 7 days. Migration branch → main → live

How Section Detection Works

Your parser reads the PHP file as a raw string. It looks for <section> tags, semantic <div> wrappers, get_template_part() calls, and PHP comments. For each chunk, it reads the CSS classes and content to classify it: a div with class hero-section containing an <h1> and a .btn link → type = hero. If confidence is low, you call an LLM to classify it.

Classes You Need to Build

8 new PHP classes + modifications to 8 existing ones. Here's exactly what each class does, in terms you understand.

Class Node/Python Equivalent What It Does Key Methods
HomepageDetector NEW A file resolver that checks WP config + filesystem to find the homepage template Checks front-page.php existence, reads WP options for static page, checks for page builders (Elementor/BB). Returns the file path + detection method. detect() → {template_path, method, page_id, page_builder}

Uses regex to find <section> tags, <div> wrappers with semantic classes, get_template_part() calls. Returns array of section boundaries with start/end line numbers. parse($file_path) → [{index, type, confidence, start_line, end_line, classes}]
SectionItemMapper NEW A data transformer that maps extracted HTML content → ACF field names Takes raw content items (headings, images, links) from a section and assigns each to the correct ACF sub-field name. E.g., the <h1> in a hero section → hero.heading. map($section_type, $items) → [{field_name, value, type}]
SectionSchemaRegistry NEW A JSON schema definition (like Zod schemas) that defines what fields each section type has Defines all 13 layout types with their sub-fields. Hero has: heading, subheading, bg_image, cta_text, cta_link, etc. Services has: heading, items (repeater with title, description, image, link). get_layout($type) → ACF layout definition array
HomepageCommand NEW CLI A CLI command handler (like a Click command or Commander action) Orchestrates P1 + P2: calls HomepageDetector → SectionParser → TemplateScanner → outputs JSON. __invoke($args, $assoc_args) — WP-CLI entry point
ApplyHomepageCommand NEW CLI A code generator (like scaffolding in Next.js or Django) Creates .bak, generates new front-page.php with FC loop, generates section partials in template-parts/sections/. __invoke()
SeedHomepageCommand NEW CLI A database seed script (like prisma db seed) Reads the scan JSON, calls update_field() for each section. Sideloads images. Creates CPT posts. __invoke()
VerifyHomepageCommand NEW CLI A test runner (like Jest/pytest assertions) Reads seeded data from DB, compares against scan JSON. Checks: counts match, content matches, images exist, links valid. __invoke()

Section 08

Gotchas & Landmines

Things that will waste your time if you don't know about them upfront.

🚨 PHP Is Not Like Node

  • ✗ No persistent process. Every HTTP request boots WordPress from scratch. No state between requests. No event loop. Think of it as a serverless function that runs and dies.
  • ✗ No package manager in the Node sense. You don't composer require for WordPress plugins. Plugins are folders dropped into wp-content/plugins/. Dependencies are checked at runtime.
  • ✗ Arrays are weird. PHP arrays are ordered maps (like JS objects + arrays combined). $arr = ['key' => 'val'] is an associative array. ACF heavily uses nested associative arrays.
  • ✗ Strings are double-quoted vs single-quoted. "$var" interpolates. '$var' is literal. Use single quotes unless you need interpolation.
  • ✗ Error handling is painful. PHP doesn't throw exceptions by default — many functions return false on failure. Always check return values.

⚠️ WordPress-Specific Traps

  • ✗ update_field() vs update_post_meta() — ACF has its own data format. Always use ACF functions (update_field, get_field) NOT raw WordPress meta functions. ACF stores metadata about field types that raw WP functions skip.
  • ✗ Flexible Content data format. FC stores as a flat array in wp_postmeta with a specific naming convention. Don't try to manually construct it — use update_field('homepage_sections', $rows).
  • ✗ Image fields need attachment IDs, not URLs. ACF image fields store the WP attachment post ID (an integer). You must media_sideload_image() to download the image first, then get its ID.
  • ✗ WP-CLI runs in a special context. Some WP functions aren't available in CLI mode. Always check if functions exist before calling them. wp_get_current_user() may not work in CLI.
  • ✗ Caching. WordPress and ACF cache aggressively. After seeding data, you MUST call wp_cache_flush() and acf_flush_field_cache() or the frontend won't show new data.
💀

The #1 Bug You Will Hit

Double tel:tel: links. The original PHP might have <a href="tel:555-1234">. Your rewriter might generate <a href="tel:<?php echo get_sub_field('phone'); ?>">. If the seeded value already contains tel:, you get tel:tel:555-1234. Strip protocols from stored values. The verification step specifically checks for this bug.

💀

The #2 Bug You Will Hit

Flexible Content row order matters. The array you pass to update_field('homepage_sections', $rows) determines the display order. If your parser detects sections out of order (because of includes or dynamic PHP), the homepage sections will render in the wrong order. Always preserve the original order from the PHP file.

PHP Syntax Cheat Sheet for You

Quick reference since you've never written PHP:

$variable — all vars start with $    -> — object method call (like . in JS)    => — key-value in arrays (like : in JS objects)    :: — static method call    . — string concatenation (not +)    === — strict equality (same as JS)    foreach ($arr as $key => $val) — like for (const [key, val] of Object.entries(arr))

// Loop through Flexible Content rows
if (have_rows('homepage_sections')):
  while (have_rows('homepage_sections')): the_row();
    $layout = get_row_layout();  // "hero", "services", etc
    $heading = get_sub_field('heading');
    $image = get_sub_field('background_image');

// Seed Flexible Content data
$sections = [

You've Got This

The plugin is fundamentally a text-processing pipeline: read PHP files as strings → parse them into structured data → generate new PHP files from templates → write data to a database.
